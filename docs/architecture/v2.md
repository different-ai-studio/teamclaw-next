## 1. 产品结构

v2 把过去"一个 Tauri 应用"拆成了**客户端 / Daemon / 后端服务**三层：

| 角色 | 形态 | 说明 |
|---|---|---|
| **TeamClaw Desktop** | Tauri 桌面应用（macOS / Linux / Windows） | 人类客户端。三栏聊天 UI、编辑器、工作区。 |
| **TeamClaw Mobile** | iOS / Android（Android 规划中） | 人类客户端的移动形态，参与 agent network。 |
| **amux daemon** | 独立后台进程 / 可单独 CLI 安装 | Agent host。承载 opencode / codex / claude 等 ACP agent。 |
| **EMQX** | 托管 MQTT broker | 实时消息总线，承载 session 内 actor 通信。 |
| **Supabase** | 托管 Postgres + Auth + Realtime | 身份、团队、会话、消息持久化。 |

**安装关系：**

- 装 TeamClaw Desktop ⇒ 自动安装 amux daemon companion service + opencode（开箱即用，本机即是 agent host）
- 服务器场景 ⇒ 单独安装 amux daemon CLI（无 GUI，纯 agent host）
- 移动客户端 ⇒ 不带 daemon，只是接入 agent network 的人类参与者

**约束：** 同一 user 同一 team **只能有一台 Desktop 在线**（= 一个 daemon = 一个 agent identity）。新 Desktop 登录会强制踢掉旧 Desktop。多设备并存仅对被动客户端（Mobile / Web）开放——它们不承载 daemon。详见 §4.5 与 §5.1。

## 3. 组件与职责

### 3.1 TeamClaw Desktop / Mobile（人类客户端）

- **职责：** UI、编辑器、工作区管理、本地文件读写；登录 Supabase；订阅自己参与的 session 的 MQTT 主题。
- **不做：** 不直接 spawn agent 进程；不维护会话权威状态。
- **桌面特有：** 携带 amux daemon 二进制，并由安装器注册为独立 companion service 自动起停；编辑器（Tiptap / CodeMirror）、终端、文件树。
- **移动特有：** 轻量人类客户端；通过 MQTT 加入 session，对话与查看，**不承载 agent**。

### 3.2 amux daemon（agent host）

- **职责：** 用 zed ACP 协议管理子进程（opencode / codex / claude）；监听属于本机 agent 的 MQTT topic；把 ACP 流转译为协议消息推到 EMQX；从 Supabase 拉取/写入会话与消息。
- **生命周期：** 一个 daemon = 一个 agent 身份；daemon 在线 ⇒ agent 在线。
- **部署方式：** 跟随桌面安装 / 服务器独立 CLI；与 TeamClaw GUI **解耦**，没有 daemon 也能跑（只是没 agent 可用）。
- **关键边界：** daemon 只校验本机 `workspace_path`；不做跨机文件同步。

### 3.3 EMQX

- **职责：** 实时消息广播。每个 session 一个 topic（`amux/{team_id}/session/{session_id}/live`），所有 actor（人 + agent）都在同一个 topic 里收发协议消息。
- **不做存储：** 持久化交给 Supabase。EMQX 仅承担"在线即时分发"。

### 3.4 Supabase

- **职责：** 单一权威状态存储 + Auth。客户端用 Postgres REST / Realtime 读写实体；JWT 用于 MQTT 鉴权。
- **不承担：** agent 调度、消息排队、文件存储（文件继续走 OSS）。

### 3.5 后端 Provider 演进

当前 v2 把 Supabase 当作权威后端：Auth、团队/成员、actor、session、message、invite、agent runtime、RPC、RLS 都落在 Supabase 上。这个选择让 Phase 1 更快跑通多人、多 agent、mobile/headless 和 daemon resume，但它不是长期的架构中心。长期目标是让 Supabase 成为一个官方 backend provider，而不是 TeamClaw 的唯一后端。

目标边界：

| Provider 面 | 现有 Supabase 能力 | 抽象后的 TeamClaw 语义 |
|---|---|---|
| Auth | GoTrue session / refresh token / JWT | 当前用户、token 刷新、登录/退出、provider-scoped credential |
| Team directory | `teams`, `team_members`, `actors`, `agents` + RLS | team、member、agent actor 的权威目录 |
| Sessions | `sessions`, `session_participants`, `messages` | 创建/列出会话、参与者、消息历史、乐观写入确认 |
| Invite / ACL | SQL RPC + RLS + `agent_member_access` | claim invite、检查 actor 是否能加入/驱动 agent |
| Runtime state | `agent_runtimes` + cursor 字段 | daemon/runtime 注册、presence 桥接、last-processed cursor |
| Attachments | Supabase Storage `attachments` bucket | 上传附件、返回可落库 object ref、按 session 权限读取 |
| Realtime | Phase 1 主要走 EMQX；Supabase 只作状态/补偿 | provider 不直接决定 session live transport |

#### 演进原则

1. **先抽 TeamClaw 领域接口，再接替代后端。** 不把 PocketBase、TrailBase 或 SQLite 的 API 形状泄漏进 UI / daemon / gateway。
2. **Supabase adapter 继续是第一实现。** 第一阶段只移动边界，不改变线上行为、不改 schema、不引入新后端。
3. **实时 transport 独立于数据 provider。** EMQX/MQTT 的 session live 语义保持独立；backend provider 负责权威状态、补偿读取、权限检查和 cursor。
4. **本地开源路径优先。** 第二个实现优先考虑 `local-sqlite` 或 TrailBase，用来验证 TeamClaw 领域接口是否足够中性。

#### 推荐路线

| 阶段 | 目标 | 结果 |
|---|---|---|
| 1. Provider boundary | 把 Supabase 调用收敛到明确接口后面，仍只运行 Supabase | 降低耦合；为替代后端留出插槽 |
| 2. Local / TrailBase adapter | 选择 `local-sqlite` 或 TrailBase 做第二实现 | 证明 TeamClaw 可以不依赖托管 Supabase 跑起来 |
| 3. PocketBase adapter | 在边界稳定后适配 collection/rule 模型 | 给轻量自托管用户一个成熟、易管理选项 |
| 4. Plain Postgres adapter | 面向高级自托管 / 企业部署 | 保留 SQL/RLS 类部署的迁移路径 |

TrailBase 适合作为较早候选，是因为它的形状更接近 Supabase 替代：single executable、SQLite/Rust、auth、REST/record API、realtime、admin UI、custom endpoint，且能更自然承接现在的 SQL/RPC 思路。PocketBase 更成熟、更易上手，但 collection/rule/hook 模型会更强地影响抽象设计，适合在 provider 契约稳定后再接。

#### 第一阶段难度评估：中等偏大，但可分块

第一阶段不是“换数据库”，而是把 Supabase 从散落依赖收敛成 adapter。难点主要来自调用面广，而不是单点技术风险：

- **Daemon Rust 侧：中等。** `apps/daemon/src/supabase/client.rs` 已经把大量 HTTP/RPC 封装在一个 client 里，`crates/teamclaw-gateway/src/supabase_store.rs` 也已经有一个接近领域语义的 `ChannelStore` trait。这说明 gateway 路径已有边界雏形。主要工作是把 `SupabaseClient` 拆成 `AuthBackend`、`SessionBackend`、`MessageBackend`、`RuntimeBackend`、`AttachmentBackend` 等 trait，并让现有 Supabase client 实现它们。
- **Frontend TS 侧：中等偏大。** `packages/app/src/lib/supabase-client.ts` 是单例，session list、auth store、sync helpers、telemetry/team config 等直接使用 `supabase-js`。这里需要先建 `backend-client` facade，再逐步把 store 改成领域方法调用。因为 UI 仍运行 Supabase，迁移可以按 store 分批做。
- **Schema/RPC 侧：低到中等。** 第一阶段不改 Supabase schema，只给现有 RPC 命名和参数形状建立 provider contract。真正困难会出现在第二实现，因为 RLS、trigger、Postgres function 需要被重新表达。
- **移动端：中等。** iOS/Android 当前直接绑定 Supabase SDK。第一阶段可以先在 desktop/daemon 建边界，移动端跟随到 Phase 2，否则会拖慢主线。
- **测试：中等。** 现有 mock Supabase 和 pgTAP 仍保留；需要新增 adapter contract tests，用同一组行为测试跑 Supabase adapter mock。第二实现出现后再复用这些 contract tests。

建议拆分：

1. 在 Rust 侧先把 gateway 已有 `ChannelStore` 改名/泛化为 backend-facing domain store，错误类型从 `Supabase` 改为 `Backend`。
2. 在 daemon 内新增 provider traits，让 `SupabaseClient` 原样实现；调用方依赖 trait object / generic，而不是具体 `SupabaseClient`。
3. 在 frontend 新增 `BackendClient` facade，先只包 auth、session list、message history、send persist 四条主链路。
4. 保持 `services/supabase/` migration 和 pgTAP 作为 Supabase adapter 的 schema contract，不因为抽象而删除。
5. 等主链路全部走 provider facade 后，再评估 local-sqlite / TrailBase 的第二实现。

粗略工作量：如果只做 Phase 1 provider boundary，不接第二后端，约 **1-2 周** 可以把 daemon/gateway 主链路收敛；加上 frontend 主链路约 **2-3 周** 更现实。若同时覆盖 iOS/Android 和所有 telemetry/team-config 支路，则会接近 **4-6 周**。真正接 TrailBase/PocketBase 是另一个阶段，不应混进第一阶段。

## 4. 协议与数据流

### 4.1 三种通道，各司其职

| 链路 | 协议 | 用途 |
|---|---|---|
| Desktop / Mobile / daemon ↔ EMQX | MQTT | **第一阶段唯一实时通道**：session 消息、agent 控制、runtime state 都先走 broker |
| 各方 ↔ Supabase | HTTPS (PostgREST) | 实体 CRUD、持久化 message |
| Daemon ↔ Agent | zed ACP（stdio） | 与 opencode / codex / claude 子进程通信 |

> 第一阶段先不做本地 IPC 短路。即使 Desktop 和 daemon 在同一台机器上，也统一通过 MQTT topic 通信，把协议、权限、runtime 生命周期和 mobile/headless 路径先跑通。IPC / Unix socket 只作为后续性能与离线优化。

### 4.2 一条消息的完整路径

以"用户在 Desktop 里 @某本机 agent 提问"为例：

1. Desktop → **Supabase HTTP**：插入 `messages` 行（用户提问，持久化）
2. Desktop → **EMQX MQTT**：publish `message.created` envelope 到 `amux/{team_id}/session/{session_id}/live`
3. 参与该 session 的 daemon 都订阅同一 topic；目标 agent 所在 daemon 收到 envelope 后，根据 `mention_actor_ids` 决定是否唤起本机 runtime
4. Daemon → **zed ACP**：转发给目标 agent 子进程
5. Agent 流式产出 token → Daemon：
   - Daemon publish 流式 delta / runtime event 到同一 session topic
   - Desktop / Mobile 订阅该 topic 后实时渲染
6. Agent 完成 → Daemon → **Supabase HTTP**：写入完整 message 行
7. 后续打开 session 的客户端：历史从 Supabase 拉；实时事件继续走 MQTT

**核心不变量：** 流式期间显示来源 = transport 推来的 delta 累积；完成后显示来源 = Supabase `messages.content`。两者**不混用**（详见根 `CLAUDE.md` Streaming Architecture 一节）。

### 4.3 协议消息格式

线协议采用 protobuf（与 amux 上游对齐），参见 spec 中 `teamclaw.proto`。所有 actor 共用同一组消息类型，**人与 agent 在协议层等价**。

#### Actor type 边界映射

Supabase 的权威枚举只有两类：`member` / `agent`。历史 proto 和本地缓存仍可能出现 `human`、`personal_agent`、`role_agent`。所有跨边界读写必须走同一个映射规则，不允许 UI / daemon / sync 各自手写判断。

| 来源值 | 规范化后 | 说明 |
|---|---|---|
| `member` | `member` | DB canonical human actor |
| `agent` | `agent` | DB canonical agent actor |
| `human` / `HUMAN` | `member` | proto / legacy human actor |
| `personal_agent` / `PERSONAL_AGENT` | `agent` | proto / legacy personal agent |
| `role_agent` / `ROLE_AGENT` | `agent` | proto / legacy role/team agent |

落库时只写 `member` / `agent`；发 protobuf 时再按当前 wire format 转成 `HUMAN` / `PERSONAL_AGENT` / `ROLE_AGENT`。如果一个入口无法判断 agent 是 personal 还是 role/team，默认按 `agents.visibility` 推导：`personal` → `PERSONAL_AGENT`，`team` → `ROLE_AGENT`。

### 4.4 Phase 1：统一 MQTT transport

**Phase 1 产品目标：team agent network first。** 第一阶段优先跑通多人、多设备、远端 daemon、共享 session、agent presence、权限校验和 runtime resume。即使 Desktop 与 daemon 在同一台机器上，也统一走 EMQX；单机私有 agent 的离线 / 低延迟优化不是 Phase 1 目标。后续 IPC / Unix socket 只能作为透明加速层加入，不能改变 MQTT envelope、权限、去重、cursor 和完成态落库语义。

**为什么先这样做：** v2 第一阶段的目标是把多人、多 agent、mobile/headless、runtime resume、权限检查和 Supabase 持久化的主链路跑通。统一 MQTT 可以让 Desktop、Mobile、daemon 共用同一套 topic、envelope、去重和权限逻辑，避免一开始就维护 IPC 与 MQTT 两套行为。

**为什么不是 Supabase Realtime：** 实测在多人 session × agent 流式 delta（高频小消息）场景下，Supabase Realtime 的吞吐与延迟不达标。EMQX 的 MQTT broker 在该类负载下成熟得多，且 retained / LWT / ACL 都是一等公民。代价是多一个运维面，详见 §5.4。

**当前核心思想：** 一份协议（protobuf envelope），第一阶段只有一种实时 transport：MQTT。

```
                  protobuf envelope
                          │
                ┌─────────▼──────────┐
                │  Transport trait   │
                │  send / recv       │
                └─────────┬──────────┘
                          │
                    ┌─────▼──────┐
                    │    Mqtt    │
                    │ (rumqttc)  │
                    └────────────┘
```

#### Desktop 启动时的 transport 选择（第一阶段）

```
启动 → 登录 Supabase → 连接 EMQX → 订阅当前用户可见的 session / runtime topic
  ├── 本机有 daemon  → daemon 也连 EMQX，Desktop 仍通过 MQTT 驱动它
  └── 本机无 daemon  → Desktop 仍可消费共享 session，但不能 host 本机 agent
```

#### 三种典型场景

**Case A · 单机 session（全 personal agent）**

```
┌─ My Laptop ───────────────┐      ┌──────────────┐
│  Desktop ──MQTT───────────┼─────▶│     EMQX     │
│  Daemon  ◀─MQTT───────────┼─────▶│ session live │
│    │                      │      └──────────────┘
│    ▼                      │
│  Agent                    │
└───────────────────────────┘
```

**Case B · 共享 session（混合本地 + 远端）**

```
┌─ My Laptop ──────────────────┐    ┌──────────────┐    ┌─ Teammate ──┐
│  Desktop ─────MQTT────────┼──────▶    EMQX     ◀──────┼── Daemon    │
│  Daemon  ◀────MQTT────────┼──────▶ session live ◀──────┼── Desktop   │
│     │                      │    └──────────────┘       │     │       │
│     ▼                      │                           │     ▼       │
│   Agent                    │                           │   Agent     │
└──────────────────────────────┘                        └─────────────┘
```

所有 actor 都在同一个 session topic 上收发；本机没有特殊路由。

**Case C · 无本地 daemon（Mobile / headless）**

```
Mobile ──MQTT──▶ EMQX  ◀──MQTT── 某台 daemon ──ACP──▶ Agent
```

Mobile 作为普通 actor 直连 EMQX，没有 host agent 的能力。

#### Daemon 的订阅决策

进入 session 时，daemon 根据参与者集合决定订阅策略。第一阶段不区分本机 / 远端 actor，只要 session 里有自己承载的 agent，daemon 就订阅该 session topic。

| 参与者 | MQTT 订阅 | 本机特殊路径 |
|---|---|---|
| 含本 daemon 承载的 agent | 订阅 `amux/{team_id}/session/{session_id}/live` | 无 |
| 不含本 daemon 承载的 agent | 不订阅，除非管理员 / runtime state 需要 | 无 |

这让 daemon 逻辑更简单：消费同一类 envelope，按 `message_id` 去重，按 `mention_actor_ids` 判断是否交给 ACP。

#### Fallback：daemon 崩了怎么办

- Desktop 仍保持 MQTT 连接，可以继续消费共享 session（看远端 actor 的消息）
- 本机 daemon 离线时，它承载的 agent presence 变为 offline，发给这些 agent 的消息不会被本机处理
- Daemon 恢复后重新连接 EMQX、恢复订阅，从 Supabase / `agent_runtimes.last_processed_message_id` 补 cursor 后继续处理

#### 取舍要点

- **先跑通协作闭环：** 所有端都走 MQTT，同一套 envelope、topic、权限、cursor
- **本机路径不是特例：** Desktop 不需要知道 agent 是否在本机，只需要 publish 到 session topic
- **单机体验可降级：** broker / Supabase 不可用时，本机 personal agent 也可能不可用；这是 Phase 1 为验证 team agent network 接受的产品取舍
- **去重靠 `message_id`：** MQTT 重投、重连补偿、乐观本地追加都可能造成重复，所有 envelope 带全局 `message_id`
- **后续优化再做 IPC：** 当主链路稳定后，可以给 Desktop ↔ 本机 daemon 增加 Unix socket 短路，但语义必须与 MQTT 完全一致

#### 安全模型：三层校验

MQTT 是共享实时总线，所以任何会触发 agent 行为的 envelope 都不能只相信 transport。Phase 1 采用三层防线：

| 层 | 负责什么 | 不负责什么 |
|---|---|---|
| **EMQX ACL** | 校验连接身份、JWT 是否有效、actor 是否可 publish / subscribe 指定 team/session/status topic | 不判断业务动作是否允许，例如某 actor 是否能驱动某 agent |
| **Supabase RLS / RPC** | 校验租户边界、session 参与关系、agent visibility、`agent_member_access`、message / participant 写入权限 | 不相信 MQTT payload 已经安全；不执行 agent runtime 行为 |
| **Daemon runtime guard** | 在调用 ACP 前重新校验 sender、session、mention、agent 权限、workspace path 和 cursor；拒绝不合法 envelope 并写入可观察错误 | 不绕过 Supabase 权限，也不接受客户端直接声明的“已授权”状态 |

关键规则：

- Desktop / Mobile publish 前可以做 UI 层预检查，但这只是体验优化；不是安全边界。
- EMQX 只保证 actor 不能跨 team / 跨不可见 session 乱发 topic；业务权限仍由 Supabase 和 daemon 判定。
- Daemon 收到 `message.created` / runtime control envelope 后，必须用自己的 agent actor 身份调用 `check_agent_permission(agent_id, sender_actor_id)`，并确认 sender 是该 session participant。
- Daemon 只处理 `mention_actor_ids` 中命中自己承载 agent 的消息；未命中则只作为历史 / presence 事件消费，不触发 ACP。
- Workspace path 永远只在 daemon 本机校验。远端 actor 不能通过 MQTT 声明任意路径可用。
- 所有可重放 envelope 必须带全局 `message_id` / `event_id`；daemon 以它们做去重，并用 `agent_runtimes.last_processed_message_id` 恢复 cursor。

### 4.5 在线状态与可见性（Presence）

Agent 的"在线"状态由其 daemon 的 MQTT 连接派生，不需要客户端主动 ping：

1. **上线**：daemon 连上 EMQX 后立即发布 retained message：
   ```
   topic:   actors/{agent_id}/status
   payload: { online: true, ts: <utc> }
   retain:  true
   ```
2. **下线（被动）**：daemon 注册 LWT（Last Will and Testament），EMQX 检测到 TCP 断开 / keepalive 超时会自动代发：
   ```
   topic:   actors/{agent_id}/status
   payload: { online: false, ts: <utc> }
   retain:  true
   ```
3. **订阅方**：任何客户端（Desktop / Mobile）订阅它**有权见到**的 agent status topic（权限由 §6 的 `visibility` 决定）。新打开 session 的客户端因为 retained，立即拿到当前状态，无需轮询。

**可见性的范围**由 `agents.visibility` 控制：`personal` 仅 agent owner 可见；`team` 在 team 目录中可见。注意：personal agent 仍然有 `actors` row；它只是不会出现在 `actor_directory` / Actors tab / mention / 新 session picker 这类团队目录入口中。

Human member 的在线状态必须支持**同一账号多端同时在线**：

1. **身份不按设备拆分**：同一个 Supabase user 在同一个 team 里仍然只有一个 `member` actor。Desktop、Mobile、Web 等客户端共享同一个 `actor_id`。
2. **在线按 device 拆分**：每个客户端安装 / 登录实例都有自己的 `device_id` 和 MQTT `client_id`，上线后发布 retained presence：
   ```
   topic:   actors/{actor_id}/devices/{device_id}/status
   payload: { online: true, platform: "desktop" | "mobile", ts: <utc> }
   retain:  true
   ```
3. **下线按 device 生效**：每个 MQTT 连接注册自己的 LWT；某台手机断线只会把该 `device_id` 标为 offline，不影响同账号的 Desktop。
4. **UI 聚合在线态**：member actor 的总体在线 = 任意一个 device 在线。需要展示细节时再展开为 "Desktop 在线 / iPhone 在线"。

因此，`actor_id` 表示"谁在说话"，`device_id` 表示"这次从哪台设备连上来"。消息作者仍写 `sender_actor_id`，不要把同一用户的多台设备拆成多个 member actor。

**Desktop 例外（单设备独占）：** 同一 user 同一 team 只能有一台 Desktop 在线——因为 Desktop 携带 daemon、daemon 承载 agent identity。新 Desktop 登录时旧 Desktop 收到 logout 信号并停止 daemon；旧设备 UI 显示"已在另一台机器登录"。设备丢失 / 重装场景走邮件 + 2FA 强制踢掉路径。**多端并存仅适用于 Mobile / Web 等不承载 daemon 的被动客户端**——它们各自有 `device_id` 但共享同一 member actor。

## 5. 部署形态

### 5.1 桌面 bundle（默认）

```
TeamClaw.app
├── Tauri 主进程（UI + Rust 命令）
└── bundled tools:
    ├── amux daemon installer / binary
    └── opencode（默认 agent runtime）
```

- 安装即可用，零配置接入 agent network。
- 首次启动会把 daemon 注册为独立 companion service：launchd（macOS）/ systemd user service（Linux）。它随 Desktop 安装，但运行时不耦合 GUI，不作为 Tauri 子进程参与会话协议。
- Windows v1 暂走手动安装路径（spec Q6）。
- **登录策略：** 同一 user 同一 team 单 Desktop 独占；新设备登录会强制踢掉旧 Desktop 并停止其 daemon（详见 §4.5）。
- **后端 URL 分发：** EMQX broker URL、Supabase URL / anon key 通过应用配置文件下发——构建时打包默认值，运行时支持远端覆盖。daemon 与 Desktop 共用同一份配置；URL 变更通过下次启动 fetch 应用，不需要重发客户端。

### 5.2 服务器 daemon（headless agent host）

- 单独 `amuxd` 二进制；`amuxd init` 完成 Supabase 配对与 BYOK 密钥；以 systemd 常驻。
- 用途：把某台机器变成"专职 agent 工位"；多人共享一台 GPU/工作站；CI 节点上跑 agent。
- 无 GUI，无桌面 UI 资源。

### 5.3 移动客户端（接入端）

- 不带 daemon，**只是 session 的另一个人类参与者**。
- 通过 Supabase Auth 登录；订阅自己参与的 session 主题；在手机上 @ 远端 agent。
- 文件类操作受限（移动平台沙盒），代码编辑等深度功能仍在桌面。

### 5.4 部署拓扑（后端栈）

| 组件 | 形态 | 责任边界 |
|---|---|---|
| **EMQX** | **自托管** | 团队自托管 broker；JWT 鉴权对接 Supabase；ACL 按 team_id / session_id scope；retained + LWT 承载 agent presence；高可用、TLS 轮换、容量规划由团队 SRE 负责 |
| **Supabase** | 托管 | Postgres + Auth；Realtime 在 Phase 1 不依赖（性能不达标，见 §4.4）；migration 走 Supabase CLI |
| **客户端配置** | 应用配置文件 + 远端覆盖 | broker URL、Supabase URL / anon key 构建时打包默认值，运行时可被远端配置覆盖；客户端启动 fetch，不需重发版本即可搬迁后端 |

自托管 EMQX 的代价：broker 高可用、TLS 证书轮换、JWT 续期对接、ACL 规则维护、容量规划全部是团队 SRE 责任。Supabase Realtime 已经在拓扑里但 Phase 1 不依赖——保留作为 Phase 2 / 降级路径的可选项。

## 6. 数据模型（Supabase）

> 命名约定：**Actor = 伞型概念**；`Member` / `Agent` 是 `actor_type` 的两种子类型。当前表设计使用 `member` / `agent`，历史 proto / 本地缓存里出现的 `human`、`personal_agent`、`role_agent` 需要在边界层兼容。

边界兼容规则见 §4.3 “Actor type 边界映射”。DB、RLS、Supabase view、local-cache 的规范值都必须保持 `member` / `agent`，避免把 wire-format 历史枚举扩散进权威数据层。

Supabase 是 v2 的权威状态层。下面是产品语义上的表结构设计；实际 migration 应保持这些列名与约束，客户端 / daemon 当前已经按这些名字读写。

### 6.1 核心表

#### `teams`

团队是所有协作数据的最外层租户边界。

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | `uuid primary key` | team id |
| `name` | `text not null` | 团队名 |
| `owner_actor_id` | `uuid references actors(id)` | owner actor；避免直接绑 auth user |
| `created_at` / `updated_at` | `timestamptz not null` | 审计时间 |

#### `actors`

Actor 是人和 agent 的统一目录。UI、协议、session participant 都只拿 `actor_id`。

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | `uuid primary key` | actor id；MQTT/protobuf 里的 `actor_id` 同源 |
| `team_id` | `uuid references teams(id)` | 租户边界 |
| `actor_type` | `text not null` | `member` / `agent`；历史代码里也可能读到 `human`、`personal_agent`、`role_agent`，migration 需要兼容或清洗 |
| `display_name` | `text not null` | UI 展示名 |
| `avatar_url` | `text` | actor 级头像 URL；human 和 agent 都可设置，UI 没有值时再用字母头像 fallback |
| `user_id` | `uuid references auth.users(id)` | human actor 才有；agent 为 `null` |
| `member_status` | `text` | human 状态，例如 `active` / `invited` / `disabled` |
| `agent_status` | `text` | agent 状态，例如 `online` / `offline` / `disabled` |
| `created_at` / `updated_at` | `timestamptz not null` | 本地缓存 delta sync 依赖 `updated_at` |

建议暴露一个 `actor_directory` view，供客户端一次查询成员和 agent：

```
actor_directory(
  id,
  team_id,
  actor_type,
  display_name,
  avatar_url,
  member_status,
  agent_status,
  created_at,
  updated_at
)
```

#### `actor_devices`

Human 客户端的设备表。它解决"同一个账号的 Desktop 和 Mobile 同时登录、同时在线"的问题。

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | `uuid primary key` | device row id |
| `actor_id` | `uuid references actors(id)` | 所属 human actor |
| `team_id` | `uuid references teams(id)` | 冗余 team id，便于 RLS / 查询 |
| `device_id` | `text not null` | 客户端本地生成并持久化的 device id |
| `platform` | `text not null` | `desktop` / `ios` / `android` / `web` |
| `display_name` | `text` | 设备展示名，例如 `Matt's MacBook` |
| `last_seen_at` | `timestamptz` | 最近在线时间 |
| `created_at` / `updated_at` | `timestamptz not null` | 审计时间 |

唯一约束：`unique(actor_id, device_id)`。同一个 `user_id` 可以在多个客户端登录，但这些客户端都归到同一个 member actor 下。

#### `agents`

Agent 的扩展表。`actors.actor_type = 'agent'` 的行在这里有一条同 id 记录。

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | `uuid primary key references actors(id)` | agent actor id |
| `team_id` | `uuid references teams(id)` | 冗余 team id，便于 RLS / 查询 |
| ~~`device_id`~~ | — | 已废弃：agent 的 daemon 路由身份已 collapse 到 `actor_id`（`agents.device_id` 列已删），MQTT 按 `actor_id` 路由 |
| `runner` / `backend_type` | `text not null` | `opencode` / `codex` / `claude_code` 等 |
| `visibility` | `text not null default 'personal'` | 可见范围：`personal` / `team` |
| `status` | `text not null default 'offline'` | 可由 daemon heartbeat / MQTT presence 派生 |
| `owner_member_id` | `uuid not null references members(id)` | agent owner；不是 team owner/admin |
| `created_at` / `updated_at` | `timestamptz not null` | 审计时间 |

Agent owner 是产品权限里的唯一 owner。Team owner/admin 不因为团队角色自动获得 agent 权限，也不能替代 `owner_member_id`。Tauri Desktop 自带 daemon 启动自注册时，以机器名创建 `actor + agent`，默认 `visibility='personal'`，`owner_member_id=current_member_id()`。

#### `agent_member_access`

Agent 权限表。daemon 在处理远端调用时通过 `check_agent_permission(agent_id, member_id)` RPC 读取它。这里的 `member_id` 与该 human actor 的 `actors.id` 同源。

| 字段 | 类型 | 说明 |
|---|---|---|
| `agent_id` | `uuid references agents(id)` | 被访问的 agent |
| `member_id` | `uuid references members(id)` | 被授权 member actor |
| `permission_level` | `text not null` | `admin` / `prompt` / `view` |
| `granted_by_member_id` | `uuid references members(id)` | 授权人 |
| `created_at` / `updated_at` | `timestamptz not null` | 审计时间 |

唯一约束：`unique(agent_id, member_id)`。

#### `workspaces`

工作区是 agent 可运行的本机目录在云端的影子记录。云端不存文件内容，也不做跨机路径翻译。

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | `uuid primary key` | Supabase workspace id |
| `team_id` | `uuid references teams(id)` | 租户边界 |
| `agent_id` | `uuid references agents(id)` | 哪个 agent/daemon 看到的工作区 |
| `name` | `text not null` | 展示名；daemon upsert 冲突键之一 |
| `path` | `text` | daemon 本机路径；只对该 daemon 有意义 |
| `archived` | `boolean not null default false` | 软隐藏 |
| `created_at` / `updated_at` | `timestamptz not null` | 审计时间 |

唯一约束：`unique(team_id, agent_id, name)`。

#### `sessions`

Session 是一段协作对话的权威记录。

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | `uuid primary key` | session id |
| `team_id` | `uuid references teams(id)` | 租户边界 |
| `created_by_actor_id` | `uuid references actors(id)` | 创建者 |
| `title` | `text not null` | 会话标题 |
| `summary` | `text` | 会话摘要 |
| `idea_id` | `uuid references ideas(id)` | 可选关联想法 |
| `last_message_preview` | `text` | session list 快速展示 |
| `last_message_at` | `timestamptz` | session list 排序 |
| `created_at` / `updated_at` | `timestamptz not null` | 本地缓存 delta sync 依赖 `updated_at` |

索引：`(team_id, last_message_at desc nulls first, created_at desc)`。

#### `session_participants`

Session 的 actor roster。客户端 @ 列表、daemon 订阅决策都从这里派生。

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | `uuid primary key` | participant row id；客户端本地缓存也接受 `${session_id}:${actor_id}` 形式 |
| `session_id` | `uuid references sessions(id)` | 所属 session |
| `actor_id` | `uuid references actors(id)` | 参与者 |
| `role` | `text` | `owner` / `member` / `guest` 等，当前可为空 |
| `joined_at` | `timestamptz not null default now()` | 加入时间 |
| `created_at` / `updated_at` | `timestamptz not null` | 本地缓存 delta sync 依赖 `updated_at` |

唯一约束：`unique(session_id, actor_id)`。注意：当前同步代码假设这张表**没有** `team_id`，按 `session_id` scoped 拉取。

#### `messages`

消息表只存完成态 / 可回放态。流式 delta 走 transport，完成后以一条或多条 message 落库。

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | `uuid primary key` | message id；用于 MQTT 重投 / 重连补偿去重 |
| `team_id` | `uuid references teams(id)` | 冗余 team id，便于 RLS / 查询 |
| `session_id` | `uuid references sessions(id)` | 所属 session |
| `turn_id` | `text` | daemon TurnAggregator 的一轮回复关联 id |
| `sender_actor_id` | `uuid references actors(id)` | 发送者 |
| `reply_to_message_id` | `uuid references messages(id)` | 回复链，可为空 |
| `kind` | `text not null` | `text` / `system` / `work_event` / `agent_tool_call` / `agent_tool_result` / `agent_reply` 等 |
| `content` | `text not null` | 展示文本 / tool result 摘要 |
| `metadata` | `jsonb not null default '{}'` | `mention_actor_ids`、tool args/result 等结构化内容 |
| `model` | `text` | agent 回复使用的模型 |
| `created_at` / `updated_at` | `timestamptz not null` | message sync 依赖 `updated_at` |

索引：`(session_id, created_at)`、`(team_id, created_at)`。

#### `agent_runtimes`

Agent runtime 是 daemon 上一个 ACP 子进程 / 后端会话的云端影子，供 mobile/desktop 找到 live runtime 并恢复 cursor。

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | `uuid primary key` | runtime row id |
| `team_id` | `uuid references teams(id)` | 租户边界 |
| `agent_id` | `uuid references agents(id)` | 哪个 agent 承载 |
| `session_id` | `uuid references sessions(id)` | 绑定的协作 session，可为空 |
| `workspace_id` | `uuid references workspaces(id)` | 工作区，可为空 |
| `backend_type` | `text not null` | `claude_code` / `opencode` / `codex` |
| `backend_session_id` | `text` | ACP/后端自己的 session id |
| `runtime_id` | `text` | daemon 生成的短 id；MQTT runtime topic segment |
| `status` | `text not null` | `starting` / `active` / `failed` / `stopped` |
| `current_model` | `text` | 当前模型 |
| `last_processed_message_id` | `uuid references messages(id)` | daemon 消费消息的 cursor |
| `last_seen_at` | `timestamptz not null` | heartbeat / presence |
| `created_at` / `updated_at` | `timestamptz not null` | 审计时间 |

唯一约束：`unique(agent_id, backend_session_id)`；daemon 当前按这个冲突键 upsert。注意：如果 `backend_session_id` 允许为空，Postgres 允许多行 `NULL` 绕过唯一冲突；migration 必须选择其一：让 `backend_session_id` `not null`、增加 partial unique index，或改用 `(agent_id, runtime_id)` 作为非空冲突键。

#### `ideas` / `claims` / `submissions`

Idea 是跨 session 的工作卡片；claim/submission 是 actor 对 idea 的执行状态。

| 表 | 关键字段 | 说明 |
|---|---|---|
| `ideas` | `id`, `team_id`, `workspace_id`, `parent_id`, `title`, `description`, `status`, `created_by`, `archived`, `metadata`, `created_at`, `updated_at` | 想法 / 任务卡片 |
| `claims` | `id`, `idea_id`, `actor_id`, `claimed_at`, `created_at`, `updated_at` | actor 认领 idea |
| `submissions` | `id`, `idea_id`, `actor_id`, `content`, `submitted_at`, `created_at`, `updated_at` | actor 提交结果 |

**核心关系：**

```
teams 1───* actors ──0/1 agents
              └──* actor_devices
      1───* workspaces
      1───* sessions ──* messages
                    └──* session_participants ──* actors
      1───* ideas ──* claims
                 └──* submissions
agents 1───* agent_runtimes
```

**协议 ↔ DB 对齐：** MQTT / protobuf 上的 `actor_id` 必须等于 Supabase `actors.id`。Agent 的扩展属性在 `agents`，但对外身份仍是 actor id。

### 6.2 RLS / RPC 约束

- 所有表按 `team_id` 做租户隔离；没有 `team_id` 的表（如 `session_participants`）必须通过 parent session 做 RLS。
- `actor_directory` 只返回当前用户所属 team 中可见 actor：member 总是按 team membership 可见；agent 只返回 `visibility='team'` 的行。`visibility='personal'` 的 agent 仍保留 `actors` row，但不进入团队目录、mention、session picker、Actors tab。
- `actor_devices` 只允许同一个 `auth.uid()` 对应的 member actor 自己 upsert / heartbeat；team member 可以读取同 session 可见 actor 的聚合在线态，但不应默认暴露 refresh token、设备敏感信息。
- 客户端可直接插入 `sessions`、`session_participants`、`messages`，但必须满足“当前 actor 属于该 team / session”的 RLS。
- daemon 使用自己的 `SupabaseConfig.actor_id` 作为 agent actor 身份；调用 `check_agent_permission` 决定远端 member 是否可驱动本机 agent。权限规则只认 agent owner 或显式 grant：team owner/admin 不自动拥有 prompt/admin 权限。
- `claim_team_invite` 这类 onboarding RPC 是创建 actor、agent、refresh token 的唯一入口，不应让客户端绕过 RPC 直接写敏感凭据。

状态切换必须通过服务端 RPC 包成事务，避免客户端散写：

- `share_agent_to_team(agent_id)`：仅 `owner_member_id` 可调用；只把 `visibility` 改为 `team`。不改变 owner，不给 team owner/admin 自动授权，不清理既有 grant。
- `make_agent_personal(agent_id)`：仅 `owner_member_id` 可调用；把 `visibility` 改为 `personal`，删除所有非 owner 的 `agent_member_access`，并 upsert owner 的 `admin` grant。
- 历史 session / messages 不因 visibility 切换而删除。team agent 转 personal 后，过去消息仍可见；该 agent 只是不再可被团队目录发现或被非 owner prompt。

### 6.3 Daemon 不是 actor

daemon host id 只是 agent 的一个属性（"我现在被这台机器上的 daemon 承载"）。（注：agent 的 daemon 路由身份原本叫 `device_id`，现已 collapse 到 `actor_id`，MQTT topic 直接按 `amux/{team_id}/{actor_id}/...` 路由，不再有 `device/` 段。）daemon **不进 `actors` 表**，UI 上不展示，普通 member 不感知。daemon 只在两处出现：

- **凭据归属**：daemon 本机持有 LLM 凭据缓存（团队默认 key 或用户 BYOK，详见 §6.6）；agent 调用 LLM 时用它的 host daemon 的密钥
- **个人运维视图**：daemon 仅对其 owner 可见——每个用户只能在自己的设置里看到"我的 daemon（本机 / 各设备）→ 承载哪些 agent"。**admin 也看不到他人 daemon**，避免 team 内部互相观测谁在线、谁的机器跑什么。Agent 维度的运维视图（"team 内有 M 个 agent，N 个在线"）走 `agents` 表，不暴露 daemon 细节。

Agent 的在线状态来自其 host daemon 的 MQTT 连接（详见 §4.5）。

**不变量：1 daemon = 1 agent identity。** Desktop 单设备独占策略（§4.5）保证用户的桌面端最多一台 daemon 同时在线；**server daemon（§5.2）仍可造多行 agent**（每行 `actor_id` 不同），UI 显示为 "Matt's Claude (laptop)"、"Matt's Claude (prod-server-1)"——这是 power-user 路径。两个 agent 在 visibility / inbox / 历史上完全独立——v1 不做"同一逻辑 agent 跨机迁移"，详见 §9.2。

### 6.4 Agent visibility 语义

| visibility | 谁能在团队目录看到 | 谁能 prompt / 管理 | Daemon MQTT 订阅 | 默认场景 |
|---|---|---|---|---|
| `personal` | 仅 agent owner 的 Settings → Agents | 仅 `owner_member_id`；`agent_member_access` 应只保留 owner `admin` | 是（第一阶段统一 MQTT；后续可优化成本机短路） | Tauri Desktop 自带启动自注册的本机 agent |
| `team` | team 内所有 member 的 Actors / mention / session picker | agent owner，或 `agent_member_access` 中有 `prompt` / `admin` grant 的 member | 是（订阅 `amux/{team_id}/...` 下的可见 session / runtime topic） | owner 主动 Share to team 的共享 agent |

**默认装机的 opencode = `personal`。** 否则每个新用户一装 Desktop 就会有一个 agent 出现在全公司视图里，太吵。共享必须 opt-in：用户在 iOS Settings → Agents → Connected personal agents 进入详情，点击 Share to team，把 visibility 改为 `team`。这一步只改变目录可见性，不改变 `owner_member_id`，也不给 team owner/admin 特权。

**转回个人：** Actors 中的 team agent 只有 agent owner 能执行 Make personal。执行后删除所有非 owner 的 `agent_member_access`，保留 / upsert owner 的 `admin` 权限。team owner/admin 如果不是 agent owner，只是普通 member；他们不能因为团队角色把 agent 转为个人或绕过权限 prompt。

**后续离线优化：** 当主链路稳定后，`personal` agent 可以增加 Desktop ↔ 本机 daemon 的本地短路，让纯单机 session 不依赖 EMQX。第一阶段不做这个分支。

### 6.5 daemon 本地数据结构

daemon 本地只存“运行所需的缓存和凭据”，不取代 Supabase 的权威状态。默认目录是 `dirs::config_dir()/amux`；macOS 通常是 `~/Library/Application Support/amux`，Linux 通常是 `~/.config/amux`。

#### `daemon.toml`

daemon 的基础配置，路径：`<config_dir>/amux/daemon.toml`。

| 字段 | 说明 |
|---|---|
| `device.id` | 本机 daemon 的 MQTT device id |
| `device.name` | 展示给管理员看的设备名 |
| `mqtt.broker_url` | EMQX broker URL |
| `team_id` | 当前 daemon 绑定的 team，可为空 |
| `agents.claude_code.binary` | Claude Code 可执行文件名，默认 `claude` |
| `agents.claude_code.default_flags[]` | 启动 Claude Code 的默认参数 |

#### `supabase.toml`

daemon 的 Supabase 登录态，路径：`<config_dir>/amux/supabase.toml`。

| 字段 | 说明 |
|---|---|
| `url` | Supabase project URL |
| `anon_key` | Supabase anon key |
| `refresh_token` | daemon actor 的 refresh token；会随 GoTrue refresh 轮换并写回 |
| `team_id` | daemon 所属 team |
| `actor_id` | daemon 承载的 agent actor id |

这份文件是敏感文件。它让 daemon 能以 agent actor 身份调用 PostgREST / RPC。

#### `sessions.toml`（runtime store）

路径：`<config_dir>/amux/sessions.toml`。这是 daemon 进程 runtime 的恢复表，跟 Supabase `sessions` 不是同一层概念。

| 字段 | 说明 |
|---|---|
| `runtime_id` | daemon 生成的短 id；live runtime topic segment |
| `acp_session_id` | ACP 后端 session id |
| `session_id` | 绑定的 Supabase `sessions.id`，可为空 |
| `agent_type` | proto `amux.AgentType` |
| `workspace_id` | 本地 workspace store id 或 Supabase workspace id |
| `worktree` | 实际工作目录 |
| `status` | proto runtime status |
| `created_at` | Unix seconds |
| `last_prompt` | 最近一次 prompt |
| `last_output_summary` | 最近输出摘要 |
| `tool_use_count` | tool 调用计数 |

#### `workspaces.toml`

路径：`<config_dir>/amux/workspaces.toml`。

| 字段 | 说明 |
|---|---|
| `workspace_id` | daemon 本地 8 字符 workspace id |
| `supabase_workspace_id` | 对应 Supabase `workspaces.id`，可为空 |
| `path` | canonical 本机路径 |
| `display_name` | UI 展示名 |

daemon 只验证本机路径存在，不做跨机器路径同步。

#### `members.toml`

历史/本地协作 store，路径：`<config_dir>/amux/members.toml`。当前云端身份以 Supabase `actors` 为准，这个文件主要保留给 invite / legacy 路径。

| 字段 | 说明 |
|---|---|
| `members[].member_id` | 本地 member id |
| `members[].display_name` | 展示名 |
| `members[].role` | `owner` / `member` |
| `members[].token` | 本地访问 token |
| `members[].joined_at` | 加入时间 |
| `members[].department` | 可选部门 |
| `pending_invites[]` | 本地待领取邀请：`invite_token`, `display_name`, `created_at`, `expires_at`, `role` |

#### `teamclaw/sessions.toml`

路径：`<config_dir>/amux/teamclaw/sessions.toml`。这是 daemon 对 TeamClaw 协作 session 的本地 roster 缓存。

| 字段 | 说明 |
|---|---|
| `session_id` | Supabase `sessions.id` |
| `team_id` | team id |
| `title` | 标题 |
| `created_by` | 创建 actor id |
| `created_at` | 创建时间 |
| `summary` | 摘要 |
| `idea_id` | 关联 idea id，可为空 |
| `participants[]` | `actor_id`, `actor_type`, `display_name`, `joined_at` |

daemon 收到 iOS / Desktop 的 `runtimeStart(session_id=...)` 时，会从 Supabase 拉 `sessions + session_participants`，写入这个缓存，然后决定是否订阅 `amux/{team_id}/session/{session_id}/live`。

#### `teamclaw/sessions/<session_id>/messages.toml`

每个 session 的本地消息缓存。

| 字段 | 说明 |
|---|---|
| `message_id` | message id |
| `session_id` | Supabase session id |
| `sender_actor_id` | 发送 actor id |
| `kind` | `text` / `system` / `agent_tool_call` / `agent_tool_result` / `agent_reply` 等 |
| `content` | 文本内容 |
| `created_at` | 创建时间 |
| `reply_to_message_id` | 可为空 |
| `mentions[]` | mention actor ids |
| `model` | agent model，可为空 |
| `metadata_json` | tool args/result 等 JSON 字符串 |
| `turn_id` | agent turn correlation id |

#### `teamclaw/sessions/<session_id>/ideas.toml`

每个 session 的本地 idea 缓存。

| 结构 | 字段 |
|---|---|
| `items[]` | `idea_id`, `session_id`, `workspace_id`, `title`, `description`, `status`, `parent_id`, `created_by`, `created_at`, `archived` |
| `claims[]` | `claim_id`, `idea_id`, `actor_id`, `claimed_at` |
| `submissions[]` | `submission_id`, `idea_id`, `actor_id`, `content`, `submitted_at` |

#### Desktop 本地 cache（不是 daemon 权威层）

Tauri Desktop 另有 `local-cache.db`（libsql/sqlite），用于快速启动和离线展示。它镜像 Supabase 的 `actor`、`session`、`session_participant`、`message`、`idea`、`claim`、`submission`，并有 `agent_runtime_event` 和 `sync_state`。它属于客户端缓存，不应被 daemon 当作权威数据源。

### 6.6 LLM 凭据：BYOK + team 默认 key

v2 不做托管 AI 代理，但通过 team-level 默认 key 解决 "team agent 由谁付费 / 谁的 daemon 撑着 team 资源" 的问题。

| 层级 | 持有者 | 用途 |
|---|---|---|
| **Team 默认 key** | Team admin 在 Supabase 配置（envelope encryption） | 新成员加入团队时自动可用；`team` visibility 的 agent 默认走它 |
| **用户 BYOK** | 用户在本机 daemon 配置（`daemon.toml` / keychain） | 覆盖 team 默认；`personal` agent 必须走自己的 key |

**结算归属：**

- `personal` agent → 永远走 owner 的 BYOK
- `team` agent → 默认走 team key；owner 可显式切到自己的 BYOK（"我请客"模式）
- Team key 失效 / 撤销 → admin UI 告警；team agent 短时降级，personal agent 不受影响

**密钥分发：** daemon 启动时从 Supabase 拉 team key（受 RLS 保护），缓存在 `<config_dir>/amux/team-key.enc`（信封加密，机密 envelope 由 Supabase RPC 颁发）。admin 在 UI 触发轮换 → 所有 daemon 下次启动生效。撤销窗口详见 §9.8。

**Phase 1 边界：** 单 team key 共享；不做 per-agent / per-model 配额；不做 token 用量审计——运维感知由 Anthropic / OpenAI console 提供。Phase 2 可扩展为多 key、配额、用量回写。

## 7. 关键设计决策（速查）

| 决策 | 原因 | 详见 |
|---|---|---|
| Daemon 独立、不耦合 GUI | 服务器场景、agent 复用 | spec Q1 |
| 人 / agent 同一 Actor 模型 | 多人 × 多 agent 协作变成"拓扑"而非特例 | spec Q2 |
| 后端栈 = EMQX + Supabase | 实时消息走 MQTT、关系数据走 Postgres，各取所长 | spec Q3 |
| Agent 跟 daemon 走，daemon 挂 agent 挂 | v1 不做 host 迁移，复杂度可控 | spec Q4 |
| `workspace_path` 单机假设 | 不做跨机路径翻译 | spec Q5 |
| BYOK + team 默认 key | v1 不做托管 AI 代理；team 默认 key 解决 team agent 付费归属 | §6.6 |
| 硬切（big-bang）+ v2 分支 | v1 在 main 上仍可发补丁；v2 在 `v2/amuxd-architecture` 演进 | spec Q8 |
| 单 Desktop 登录独占 | 简化 agent identity 心智；多设备并存仅限 Mobile/Web | §4.5, §5.1 |
| 自托管 EMQX | Supabase Realtime 性能不达标；接受多一个 SRE 面 | §4.4, §5.4 |
| Daemon 完全隐私（含 admin） | 避免 team 内部互相观测；agent 维度的运维走 `agents` 表 | §6.3 |

## 8. 与 v1 的边界

- **改了：** session / identity / message 全部落到 Supabase；agent host 从内嵌 sidecar 变成独立 daemon；线协议改为 protobuf-over-MQTT。
- **保留：** gateway（WeCom 等渠道接入）、cron、OSS 团队同步、Tiptap / CodeMirror 编辑器、工作区 UI、本地 RAG (Tantivy)、Whisper STT。
- **不迁移历史数据：** v2 是 clean break；v1 聊天记录不自动导入。

### 8.1 上线 / 回滚策略

- **Release channel：** v2 走单独 channel（TeamClaw Beta），v1 用户主动升级；channel 合并节奏视 Beta 反馈决定，至少 1-2 个月并存
- **回滚窗口：** v2 客户端在升级后保留**只读**访问 v1 本地 cache 90 天——防 v2 重大事故时用户仍能查历史；正式回滚到 v1 版本需要用户重装
- **Schema migration：** 通过 Supabase CLI；关键唯一键变更（如 `agent_runtimes`，详见 §9.6）走 partial unique index 实现 zero-downtime
- **后端搬迁：** EMQX / Supabase URL 通过应用配置 + 远端覆盖下发（§5.4），不需要重发客户端

## 9. 已知缺口 & Phase 2 路线

v2.md 描述的是 Phase 1 架构。下列限制是有意识的取舍，**用户会感受到**，需要 Phase 2 解决：

### 9.1 Mobile-only 用户没有 agent

- **现状：** 没有 Desktop / 服务器 daemon 的用户登录 Mobile 后无 agent 可 @
- **用户体验：** onboarding 引导明确提示"需要 Mac/PC Desktop 或请 admin 配置共享 agent"，不要让 Mobile 用户卡在"@ 谁"找不到目标
- **Phase 2 方向：** managed / team 托管 daemon——admin 在云端开一台共享 daemon，mobile-only 用户也能 @ 团队 agent

### 9.2 Server daemon 仍可造多 agent identity

- **现状：** 单 Desktop 登录解决了桌面端的 agent 身份心智混乱；但用户在多台服务器装 `amuxd` 仍会产生多行 `agents`
- **用户体验：** server daemon 是 power-user 路径；UI 必须显式标注每个 agent 的 host（如 "Matt's Claude (prod-server-1)"），避免 @ 时不知 @ 哪一个
- **Phase 2 方向：** 如果产品需要"逻辑 agent + N host"，再做 agent identity 抽离；v1 不做

### 9.3 流式不变量靠"约定"

- **现状：** §4.2 末段的流式来源 / 完成态来源不混用，目前是开发纪律不是代码约束
- **风险：** v1 已经因此踩过坑（见根 `CLAUDE.md` § Streaming Architecture），review 拦不住
- **Phase 1 必须做：** 在 store / 类型系统层把 `streamingContent` 与 `messages.content` 物理分离——例如分别用不同类型 + 不允许互转的 API，让"混用"在编译期不可能

### 9.4 EMQX retained 在 broker 重启后的恢复窗口

- **现状：** retained message 是 broker memory，broker 重启 = 临时丢失 agent presence
- **缓解：** daemon 重连即重发自己的 retained（包括 LWT 配置）；窗口典型 < 5s
- **用户体验：** broker 重启窗口内 UI 显示"agent 状态未知"而不是直接"离线"，避免误导用户判断 agent 真死了

### 9.5 Actor type 边界兼容是持续 tax

- **现状：** proto wire format 仍用 `HUMAN / PERSONAL_AGENT / ROLE_AGENT`，DB / 业务层用 `member / agent`（§4.3）
- **风险：** 每个新边界都要查映射表；散落多处必出 bug
- **要求：** 转换收敛到**一个**库——Rust 一份、TypeScript 一份，全 codebase 必须用它，不允许各处手写 match。Phase 2 把 proto 升级到 `member / agent` 后这层可以删

### 9.6 `agent_runtimes.backend_session_id` NULL 唯一键

- **现状：** 当前唯一约束 `(agent_id, backend_session_id)`；Postgres 允许多行 `NULL` 绕过冲突
- **决议：** 改用 `(agent_id, runtime_id)` 作为非空冲突键。migration 阶段加 partial unique index 兼容历史行；新插入路径切到新键

### 9.7 message_id 由客户端生成（idempotency 根）

- **现状：** MQTT 重投 + 乐观本地追加 + 重连补偿三个路径都能造重复消息
- **要求：** `message_id` 由发送 actor 在客户端生成（UUIDv7，自带时间序），断网重试携带同一 id；服务端按 `id` upsert；MQTT 订阅方按 `id` 去重

### 9.8 BYOK / team key 撤销的故障窗口

- **现状：** team admin 撤销 team key → 在 daemon 下次拉取前仍可能用旧 key 调用 LLM
- **缓解：** daemon 启动 + 每 N 分钟 fetch；token 调用前可选 fast-fail 检查（Phase 2）
- **Phase 1 接受窗口：** 最长一个 daemon 重启周期；admin UI 必须明确告知"撤销后最长 N 分钟生效"
