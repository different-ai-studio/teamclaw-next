01
为什么

传统 SaaS 的天花板

🧱 标准软件的困境

一套标准功能,试图满足所有商家。结果是:大商家嫌不够、小商家嫌太重,定制要排期、要外包,演进缓慢且昂贵。每一种独特的经营方式,都被标准化磨平。

✦ TeamClaw 的主张

平台提供标准化的核心能力(客户/商品/销售/权限),再用 AI 驱动的应用生成 让每个商家基于自身场景自主创造、持续演进专属业务应用。

AI 负责降低应用创造的门槛,而不是限制应用创造的能力。
每个商家都拥有创造、部署、持续演进自身业务应用的能力。
02
总体架构

Core SaaS + Customer App 双层

⬡ Core SaaS · 统一业务底座

身份认证与权限 · 客户管理(CRM) · 商品管理 · 销售管理 · 组织与用户 · 文件与消息 · API 与事件。

统一管理企业核心数据,通过标准 API 对外开放。

React · TanStack · PostgreSQL · 多租户 tenant_id

◆ Customer App · 商家专属应用

营销系统 · 客户跟进 · 会员积分 · 预约管理 · 行业专属业务系统……

独立代码、独立数据库结构、独立业务逻辑,通过 API 调用 Core SaaS,松耦合集成。

TanStack Start · Drizzle ORM · PostgreSQL

TeamClaw Desktop(Tauri · React · TypeScript)负责 AI 生成、编辑、预览和部署应用——把"造一个 app"变成对话。
03
已锁定

四个架构决策

①

隔离模型 = 混合

逻辑独立、物理共享:每商家独立 pg schema(非独立实例)+ 独立逻辑模块,共享平台运行时;大客户可下沉独立部署。

②

AI 产物 = 真实工程代码

codegen 完整 TanStack Start + Drizzle 工程,作为可部署模块运行。创造力无上限,贴合 teamclaw "AI 写代码" 基因。

③

Core 来源 = betly 去垂直化

把现有攀岩 SaaS 抽象成通用底座,攀岩业务降级为平台上第一个 Customer App——吃自己狗粮。

④

部署底座 = Dokploy

原生 Docker Swarm 横扩 + 可下钻底座,包在自有 DeployProvider 抽象后,预留 Swarm / k3s 逃生口。

Desktop
1

AI codegen

S4 Deploy
4

一键部署

push managed git → DeployProvider → Dokploy 构建 → provision 独立 schema → 上线 <merchant>.apps.<域>。

Core SDK
5

接入核心能力

app 调 Core SaaS 客户 / 商品 / 销售 SDK,认证终端用户;支付 / 通知 / 存储 / AI 直接调,密钥在平台。

Evolve
6

持续演进

商家继续用 AI 改需求,push 即重新部署,版本可回滚——经营系统随生意一起长。

全程商家不写代码、不碰运维、不持密钥;AI 把"造一个 app"变成一段对话,平台 batteries 兜住其余一切。

    08
    架构原则

    六条不变量

    1

    稳定的底座

    Core SaaS 负责稳定的基础经营能力。

    2

    灵活的创新

    Customer App 负责灵活的业务创新与个性化。

    3

    能力即 API

    所有核心能力均通过 API 对外开放。

    4

    松耦合

    应用与核心系统保持松耦合架构。

    5

    AI 降门槛

    AI 降低应用创造门槛,而非限制创造能力。

    6

    商家自主

    每个商家都能创造、部署、持续演进自身应用。
