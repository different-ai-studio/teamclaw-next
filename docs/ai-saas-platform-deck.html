行业标杆 · 巨人的架构

Salesforce Platform

1999 年以 “No Software” 开创多租户 SaaS,如今是全球最大的企业 SaaS,服务数百万组织。它的架构定义了“在平台上配置出你的应用”这一代范式。

两根支柱:多租户共享 + 元数据驱动。客户的对象、字段、流程乃至 Apex 代码,都在同一套共享平台内,靠元数据与专有运行时解释执行。

定制不出围墙:声明式拖拽 + 编程式 Apex,产物是平台托管的元数据应用,而非每租户独立的真实软件。

一套共享平台、元数据建模、专有运行时与语言——上一代企业 SaaS 的集大成者。

03
转机 · 为什么是现在
↗新的转机,来了

LLM 已经强到,普通人也能造出复杂的软件。

软件创造的门槛,正从「会编程」塌缩到「会表达」——这是过去三十年没有过的拐点。

🧱 传统 SaaS 的天花板

标准之困:一套标准软件试图满足所有商家——大商家嫌不够、小商家嫌太重,定制要排期、要外包,演进慢且贵,每种经营方式都被磨平。

实施与支持之重:真正贵的是落地——员工不懂系统、人员不断流动,每换一拨人就得重新培训;要让软件一直被用起来,就得不停地支持和维护。

✦ 我们的主张

平台提供标准化的核心能力(客户/商品/销售/权限),再用 AI 驱动的应用生成 让每个商家基于自身场景自主创造、持续演进属于自己的经营系统。门槛交给 AI,创造力还给商家。

可是 —— 把“人人都能造软件”直接丢进企业级的数据、权限与集成里,它真能自己长对吗?
04
但是 · 企业级的复杂

光有 LLM 不够,要有一套 Harness

让“人人都能造软件”在企业里落地,难点不在写代码,而在写对、写安全、能集成、能演进。需要一套护栏与轨道,把 AI 的自由度收在“能跑且可靠”的范围内。

🔒 数据与安全

多租户隔离、权限 RLS、密钥不外泄——不能让每个人各自裸写,撞一次就是事故。

🔗 与核心系统集成

客户 / 商品 / 销售要有统一底座,各 app 共享同一份真相,而非数据各自为政。

🚀 部署与可维护

上线、回滚、监控、升级——商家不该碰运维,平台替它兜住。

🛡 稳定的轨道

模板 + SDK + 约束,让 AI 生成的应用天然正确、可集成、可演进。

这套 harness = Core SaaS(稳定能力)+ 平台 batteries + codegen 护栏。Betly 就是这套 harness —— 下面讲它怎么搭。

已锁定

四个架构决策

①

隔离模型 = 混合

逻辑独立、物理共享:每商家独立 pg schema(非独立实例)+ 独立逻辑模块,共享平台运行时;大客户可下沉独立部署。

②

AI 产物 = 真实工程代码

codegen 完整 TanStack Start + Drizzle 工程,作为可部署模块运行。创造力无上限,贴合 betly "AI 写代码" 基因。

③

Core 来源 = betly 去垂直化

把现有攀岩 SaaS 抽象成通用底座,攀岩业务降级为平台上第一个 Customer App——吃自己狗粮。

④

部署底座 = Dokploy

原生 Docker Swarm 横扩 + 可下钻底座,包在自有 DeployProvider 抽象后,预留 Swarm / k3s 逃生口。
